## Часть 1. Безопасность (по убыванию серьёзности)

### 🔴 CRITICAL

**1. [ИСПРАВЛЕНО 2026-07-23] Открытый прокси `/api/[...path]` пробрасывал любой путь с админским API-ключом**
`web/app/api/[...path]/route.ts` форвардил ЛЮБОЙ `path`/`method` на бэкенд с
общим `API_KEY`, принадлежащим admin-пользователю (`seed.ts`). Единственная
защита — проверка `Origin`/`Referer`, тривиально проходимая обычным `fetch()`
с самой страницы сайта. Позволяло из консоли браузера: слить сырые API-ключи
(`/api-keys/all`), email всех пользователей (`/users`), выдать себе
безлимитную подписку без оплаты (`POST /bot-users/subscription`).
**Фикс:** явный allow-list путей (`info`, `download/video`, `download/audio`,
`download/quota`, `download/:id/status`) — всё остальное теперь `404`.
Заодно починен смежный баг: query-string вообще не пробрасывался в бэкенд
(`GET /download/quota?telegramId=` терял параметр), из-за чего заодно был
закрыт и IDOR (пункт 3 ниже) для этого пути — telegramId теперь берётся из
сессии, не от клиента.

**2. [ИСПРАВЛЕНО 2026-07-23] Необработанный reject мог уронить весь сервер**
`download.service.ts`: колбэки `complete` внутри RxJS-подписок делали
`fs.unlinkSync`/`statSync` без try/catch. `CleanupService` удалял файлы по
mtime > 1ч, не глядя в статус БД — при видео до 4ч длиной cron мог удалить
входной файл посреди конвертации → `ENOENT` → необработанный reject →
падение всего процесса Node, обрывающее скачивания ВСЕХ пользователей.
**Фикс:** (a) `main.ts` — глобальные `unhandledRejection`/`uncaughtException`
хендлеры как страховочная сетка; (b) все 3 `complete`-колбэка в
`download.service.ts` обёрнуты в try/catch с корректным переводом в
`FAILED`; (c) `CleanupService` теперь пропускает весь цикл очистки, если
есть хоть одно активное (DOWNLOADING/CONVERTING) скачивание.

### 🟠 HIGH — исправлено 2026-07-23 (коммит `1d0d160`)

**3. [ИСПРАВЛЕНО] IDOR на `GET /bot-users/:telegramId/subscription`**
Сам эндпоинт (`bot-user.controller.ts:96-108`) по-прежнему не проверяет
владение telegramId — но выяснилось, что он и не нужен как отдельный фикс:
единственный browser-reachable путь к нему — через прокси `/api/bot-users/*`,
а тот уже был закрыт allow-list'ом из критического фикса (`bot-users/*` в
список не входит, проверено — `404`). Внешний прямой путь `/bot-users/...`
(в обход Next.js) тоже проверен `curl` — `404`, nginx его никуда не роутит.
Отдельного кода не потребовалось, риск был устранён как побочный эффект.

**4. [ИСПРАВЛЕНО] Login CSRF на `POST /api/auth/telegram`**
Проверка `Origin`/`Referer` вынесена в общий `web/lib/auth/request-origin.ts`
(`isSameOriginRequest`) и подключена и сюда (раньше не было вообще), и в
`[...path]/route.ts` (раньше была локальной копией). Проверено end-to-end:
без/с чужим `Origin` → `403` до похода в бэкенд; с верным `Origin` — доходит
как раньше, без регрессий на `/api/download/quota` и `/api/info`.

### 🟡 MEDIUM — исправлено 2026-07-23 (коммит `58cd200`)

- **server**: TOCTOU в `enforceWebLimits` закрыт per-telegramId
  promise-мьютексом (`withWebLimitLock`) вокруг проверки лимита и создания
  записи — параллельные вкладки больше не могут обе пройти проверку до
  того, как первая запись появится в БД. `resolveDownloader()` теперь
  бросает `BadRequestException` на неизвестной платформе вместо тихого
  `@default(YOUTUBE)`.
- **bot**: `sessions` теперь ключом `chatId:messageId` (не просто `chatId`)
  — вторая ссылка в чате до клика по первой больше не перезаписывает сессию
  первой. `api()` — таймаут 30с через `AbortController` + безопасный парсинг
  не-JSON ответа. Годовая подписка теперь аддитивна: перед перезаписью
  `until` бот спрашивает у сервера текущий `subscriptionUntil` и прибавляет
  год к максимуму из него и "сейчас". На `SIGTERM` (обычный путь редеплоя)
  бот уведомляет все чаты с активным скачиванием вместо тишины.
- **web**: `mergeTimeseries`/`mergeErrorsTimeseries` сортируют по сырой
  ISO-дате перед рендером, не по порядку вставки в Map. `footer.tsx` читает
  `NEXT_PUBLIC_TELEGRAM_BOT_USERNAME` вместо хардкода. Новый
  `AuthProvider` (`web/lib/auth/context.tsx`) — один `/api/auth/me` на
  страницу вместо трёх независимых. Ключ дашборд-аналитики больше не
  живёт в `sessionStorage`: `POST /api/dashboard/auth` меняет его один раз
  на httpOnly-cookie, `GET /api/dashboard/[...path]` проксирует `/analytics/*`
  используя эту cookie — сам ключ в клиентском JS не появляется вообще.

## Часть 2. Рефакторинг / технический долг

**1. [ИСПРАВЛЕНО 2026-07-23, коммит `1318220`] Обобщить smoke-проверку `NEXT_PUBLIC_*` в `scripts/deploy.sh`**
Раньше проверялись только 2 конкретных ключа руками. Теперь список
генерируется из фактического использования в коде (`grep` по `web/`,
исключая `node_modules`/`.next` — при первом прогоне без исключений в
список ложно попал сам `NEXT_PUBLIC_` из комментария в `.d.ts` внутри
`node_modules`, поймано smoke-тестом до коммита), проверяется, что каждая
передана как build-arg в `docker-compose.prod.yml`, и что её значение (где
резолвится локально) реально попало в собранный бандл. Заодно заменил
`sleep 5` в проверке `server` на 15-секундный retry-цикл — фиксированная
пауза трижды за сессию давала ложный `✗` на контейнере, который на самом
деле стартовал нормально, просто чуть медленнее 5с.
