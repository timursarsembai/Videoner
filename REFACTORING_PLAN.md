# Videoner — план безопасности и рефакторинга

Составлено 2026-07-23 по итогам тщательного код-ревью (3 параллельных агента:
server/web/bot) с личной верификацией самых серьёзных находок. Обновлять по
мере фиксов — статус в начале каждого пункта.

## Часть 1. Безопасность (по убыванию серьёзности)

### 🔴 CRITICAL

**1. [ИСПРАВЛЕНО 2026-07-23] Открытый прокси `/api/[...path]` пробрасывал любой путь с админским API-ключом**
`web/app/api/[...path]/route.ts` форвардил ЛЮБОЙ `path`/`method` на бэкенд с
общим `API_KEY`, принадлежащим admin-пользователю (`seed.ts`). Единственная
защита — проверка `Origin`/`Referer`, тривиально проходимая обычным `fetch()`
с самой страницы сайта. Позволяло из консоли браузера: слить сырые API-ключи
(`/api-keys/all`), email всех пользователей (`/users`), выдать себе
безлимитную подписку без оплаты (`POST /bot-users/subscription`).
**Фикс:** явный allow-list путей (`info`, `download/video`, `download/audio`,
`download/quota`, `download/:id/status`) — всё остальное теперь `404`.
Заодно починен смежный баг: query-string вообще не пробрасывался в бэкенд
(`GET /download/quota?telegramId=` терял параметр), из-за чего заодно был
закрыт и IDOR (пункт 3 ниже) для этого пути — telegramId теперь берётся из
сессии, не от клиента.

**2. [ИСПРАВЛЕНО 2026-07-23] Необработанный reject мог уронить весь сервер**
`download.service.ts`: колбэки `complete` внутри RxJS-подписок делали
`fs.unlinkSync`/`statSync` без try/catch. `CleanupService` удалял файлы по
mtime > 1ч, не глядя в статус БД — при видео до 4ч длиной cron мог удалить
входной файл посреди конвертации → `ENOENT` → необработанный reject →
падение всего процесса Node, обрывающее скачивания ВСЕХ пользователей.
**Фикс:** (a) `main.ts` — глобальные `unhandledRejection`/`uncaughtException`
хендлеры как страховочная сетка; (b) все 3 `complete`-колбэка в
`download.service.ts` обёрнуты в try/catch с корректным переводом в
`FAILED`; (c) `CleanupService` теперь пропускает весь цикл очистки, если
есть хоть одно активное (DOWNLOADING/CONVERTING) скачивание.

### 🟠 HIGH — не исправлено

**3. IDOR на `GET /bot-users/:telegramId/subscription`**
`bot-user.controller.ts:96-108` — чужой telegramId можно запросить без
проверки владения (в отличие от `download/quota`, который уже пофикшен
выше). Любой посетитель может перебирать ID и узнавать чужой статус
подписки.

**4. Login CSRF на `POST /api/auth/telegram`**
`web/app/api/auth/telegram/route.ts` не делает `Origin`/`Referer`-проверку,
в отличие от соседнего `[...path]/route.ts`. Позволяет через cross-site POST
залогинить браузер жертвы в аккаунт атакующего (ранее полученный валидный
Telegram-payload переигрывается).

### 🟡 MEDIUM — не исправлено

- **server**: race condition (TOCTOU) в `enforceWebLimits` — параллельные
  вкладки могут пробить дневной лимит скачиваний. `resolveDownloader()`
  молча пишет `YOUTUBE` в БД при несовпадении платформы вместо ошибки
  (сейчас не проявляется, хрупко на будущее — новая платформа без
  соответствующего значения в enum `Downloaders` будет искажать аналитику).
- **bot**: `sessions` Map ключом `chatId`, не `userId` — в групповых чатах
  клик одного пользователя по чужой ссылке расходует его квоту/подписку.
  `api()` — `fetch` без таймаута может подвесить весь диалог. Годовая
  подписка при повторной покупке ДО истечения текущей не складывается, а
  перезаписывает `until` (нужно свериться с контрактом сервера). Рестарт
  бота молча теряет активные скачивания без единого сообщения пользователю.
- **web**: графики на `/dashboard` (`mergeTimeseries`/`mergeErrorsTimeseries`
  в `dashboard/page.tsx`) не сортируют точки по дате перед рендером — может
  исказить линию тренда. `footer.tsx` хардкодит `"VideonerBot"` вместо
  чтения `NEXT_PUBLIC_TELEGRAM_BOT_USERNAME`, как везде — рассинхрон при
  смене бота. `UserMenu`/`VideoInfo` дают 3 независимых одинаковых запроса
  `/api/auth/me` на одну загрузку страницы. Ключ дашборд-аналитики хранится
  в `sessionStorage` и шлётся напрямую в обход прокси — цель для XSS.

### 🟢 LOW — не исправлено

- server: мёртвый `platforms` в `lib/config.ts` (устарел, не импортируется);
  fire-and-forget без `.catch` в `info.service.ts`; несогласованная
  валидация ID (`ParseUUIDPipe` не везде).
- bot: `callback_data` парсится без runtime-валидации формата; `res.json()`
  без try/catch может протечь техническую ошибку в чат; `sessions` Map без
  TTL (в отличие от `userRequests` рядом).
- web: `JsonLd.tsx` — `dangerouslySetInnerHTML` без экранирования
  `</script>` (сейчас безопасно, ловушка на будущее); substring-матчинг
  пути в `theme-provider.tsx`; `Hero.handleDownload` не сбрасывает `loading`
  на success-пути; мёртвое поле `isDownloading` в `VideoInfo.tsx`;
  `downloadFile()` предполагает same-origin URL.

## Часть 2. Рефакторинг / технический долг

**1. Обобщить smoke-проверку `NEXT_PUBLIC_*` в `scripts/deploy.sh`**
Сейчас проверяются только 2 конкретных ключа руками. Вместо этого —
сгенерировать список фактически используемых `NEXT_PUBLIC_*` из кода
(`grep -ohr 'NEXT_PUBLIC_[A-Z_]*' web --include=*.tsx --include=*.ts | sort -u`)
и проверить, что КАЖДЫЙ передан как build-arg в `docker-compose.prod.yml` и
присутствует в собранном бандле — тогда любая будущая новая переменная
защищена автоматически, а не только сегодняшние.

**2. Распространить `scripts/deploy.sh` на `bot`**
Сейчас скрипт покрывает `server`/`web`; `bot` всё ещё деплоится вручную ad
hoc командами. Общий безопасный путь (build+rm+up+проверка логов) должен
быть один для всех трёх сервисов.

**3. Схлопнуть дублирующуюся NULL/OTHER-группировку ошибок в один хелпер**
Один и тот же баг (NULL и enum `'OTHER'` — разные группы в SQL) чинился
дважды в `analytics.service.ts` (`errors()` и `errorsTimeseries()`) — два
отдельных куска кода с одинаковой логикой. Вынести в общий
`mergeErrorCategory()`.

**4. Задокументировать/тестом закрепить тройное дублирование "какое качество платное"**
`isPaidQuality` (bot.ts) / `isPaidVideoQuality` (server, web) — порог 720p
продублирован в трёх местах на трёх рантаймах (сервер авторитетен, это не
дыра безопасности, но риск рассинхрона при будущей правке). Минимум —
комментарий-ссылка на два других файла + один интеграционный тест.

**5. Staging-контур вместо тестирования прямо в проде**
Самый дорогой пункт, но именно его отсутствие объясняет, почему баги с
Turnstile/OAuth ловились только вживую пользователем. Хотя бы отдельный
поддомен (`staging.videoner.download`) с той же docker-compose конфигурацией
для браузерозависимых изменений перед прод-деплоем.

**6. Разбить god-функцию/god-объекты**
- `download.service.ts` `downloadVideo()` (~200 строк, 3 уровня вложенных
  callback-подписок) → вынести `runDownload()`/`handleConversion()`
  отдельными приватными методами.
- `ytdlp.service.ts` (815 строк, 4-5 зон ответственности в одном классе) →
  разделить на `YtdlpProcessService` (spawn/семафоры/throttling) и
  `YtdlpFormatService` (info/playlist/парсинг форматов).
- `bot/src/bot.ts` (494 строки, всё на одном уровне) → вынести хендлеры
  `message:text`/`callback_query:data` в отдельные модули/сервисы.
